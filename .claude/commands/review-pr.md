以下の手順でセルフレビューを実施し、PR を作成してください。

## ステップ1：変更内容の把握

対応する Issue 番号を確認してください（不明な場合はユーザーに確認）。

```bash
git log main..HEAD --oneline
git diff main..HEAD --stat
```

## ステップ2：スコープ外変更チェック

Issue のタイトル・やること と `git diff main..HEAD` を照合し、以下を確認してください：

- [ ] Issue のスコープ外のファイルが変更されていないか
- [ ] 意図しない変更（デバッグコード、無関係なリファクタ等）が混入していないか

スコープ外の変更が見つかった場合はユーザーに指摘し、別 Issue・別 PR に分けることを提案してください。

## ステップ3：セルフレビューループ（LGTM になるまで繰り返す）

以下をループしてください：

### 3-1. コードレビューを実施する

`git diff main..HEAD` および `git diff`（未コミット変更）の差分全体を分析し、以下の観点でレビューしてください：

- コードの正確性・バグの有無
- プロジェクトの規約・スタイルとの一貫性
- 不要なコード・コメント・デバッグ痕跡の混入
- セキュリティ上の問題
- 改善すべき点

### 3-2. 判定

- **指摘なし → LGTM**：ステップ4に進む
- **修正すべき指摘あり**：3-3 へ

### 3-3. 修正してコミット

指摘箇所を修正し、`/commit` の手順でコミットしてください。その後 3-1 に戻る。

> ループを抜ける条件：3-2 で「指摘なし」と判定できたとき。

## ステップ4：PR description のドラフト

PR テンプレートに沿って description を生成してください：

```
## 変更内容

（変更の概要を箇条書きで）

## 対応 Issue

Closes #XX

## 影響範囲

（変更によって影響を受ける機能・ファイル）

## 確認項目
- [ ] lint 通過（PostToolUse Hook が通過済みなら `[x]` にする）
- [ ] typecheck 通過（PostToolUse Hook が通過済みなら `[x]` にする）
- [ ] test 通過
```

## ステップ5：ユーザーに確認

セルフレビューの結果サマリーとステップ4のドラフトをまとめてユーザーに提示し、承認を得てください。

## ステップ6：PR 作成

承認後、以下のコマンドで PR を作成してください。`--body` にはステップ4で生成した実際のテキストをそのまま展開して渡すこと：

```bash
gh pr create \
  --repo reiya0104/kanji-learning \
  --title "<prefix>: <変更内容>" \
  --body "$(cat <<'EOF'
（ステップ4で生成した description を、ここに直接貼り付ける）
EOF
)"
```

PR 作成後、URL をユーザーに提示してください。
