以下の手順でセルフレビューを実施し、PR を作成してください。

## ステップ1：変更内容の把握

対応する Issue 番号を確認してください（不明な場合はユーザーに確認）。

```bash
git log main..HEAD --oneline
git diff main..HEAD --stat
```

## ステップ2：スコープ外変更チェック

Issue のタイトル・やること と `git diff main..HEAD` を照合し、以下を確認してください：

- [ ] Issue のスコープ外のファイルが変更されていないか
- [ ] 意図しない変更（デバッグコード、無関係なリファクタ等）が混入していないか

スコープ外の変更が見つかった場合はユーザーに指摘し、別 Issue・別 PR に分けることを提案してください。

## ステップ3：セルフレビューループ（LGTM になるまで繰り返す）

以下をループしてください：

### 3-1. コードレビューを実施する

ビルトインスキル `/review` を使ってレビューしてください。

- PR が作成済みの場合：`/review <PR番号>`
- PR 未作成の場合：`/review` を実行し、`gh pr list` の代わりに `git diff main..HEAD` および `git diff` の差分を分析対象にしてください

レビュー結果は以下の3レベルに分類して出力してください：

| レベル | 対応方針 |
|--------|---------|
| **Critical** | 必ず対応する（バグ・セキュリティ・規約違反など） |
| **Warning** | できれば対応する（品質改善・潜在的問題など） |
| **Info** | YAGNI にならず、より良くなるなら対応する（nit・imo など） |

### 3-2. 判定

- **Critical がない → LGTM**：ステップ4に進む（Warning/Info は任意対応）
- **Critical あり**：3-3 へ

### 3-3. 修正してコミット

Critical の指摘箇所を修正し、`/commit` の手順でコミットしてください。Warning/Info は判断して対応する。その後 3-1 に戻る。

> ループを抜ける条件：3-2 で Critical が0件になったとき。

## ステップ4：PR description のドラフト

PR テンプレートに沿って description を生成してください：

```
## 変更内容

（変更の概要を箇条書きで）

## 対応 Issue

Closes #XX

## 影響範囲

（変更によって影響を受ける機能・ファイル）

## 確認項目
- [ ] lint 通過（PostToolUse Hook が通過済みなら `[x]` にする）
- [ ] typecheck 通過（PostToolUse Hook が通過済みなら `[x]` にする）
- [ ] test 通過
```

## ステップ5：ユーザーに確認

セルフレビューの結果サマリーとステップ4のドラフトをまとめてユーザーに提示し、承認を得てください。

## ステップ6：PR 作成

承認後、以下のコマンドで PR を作成してください。`--body` にはステップ4で生成した実際のテキストをそのまま展開して渡すこと：

```bash
gh pr create \
  --repo reiya0104/kanji-learning \
  --title "<prefix>: <変更内容>" \
  --assignee "@me" \
  --body "$(cat <<'EOF'
（ステップ4で生成した description を、ここに直接貼り付ける）
EOF
)"
```

PR 作成後、URL をユーザーに提示してください。
