以下の手順でコミットを作成してください。

## ステップ1：変更内容の確認

```bash
git status
git diff --staged
git diff
```

staged な変更がなければ `git add` するファイルを選んでください。
`.env` や認証情報を含むファイルは絶対に含めないこと。

## ステップ2：domain 層チェック

`src/domain/` 以下のファイルが変更に含まれる場合、以下を確認してください：

- [ ] 対応するテストが更新・追加されているか
- [ ] 復習ルール（`src/domain/review.ts`）を変更した場合、`docs/adr/0003-review-rule.md` を更新したか

未対応の項目があればユーザーに指摘し、コミット前に対処してもらうこと。

## ステップ3：コミットメッセージの作成

変更内容から適切な prefix を選ぶ：

| prefix | 用途 |
|--------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `chore` | ビルド・設定・依存関係の変更 |
| `docs` | ドキュメントのみの変更 |
| `test` | テストのみの変更 |
| `harness` | 開発ハーネス・CI・ツールの整備 |

**件名**（1行目）：72字以内
```
<prefix>: <変更内容を日本語で簡潔に>（#Issue番号）
```

**本文**（3行目以降）：「なぜこの変更をするのか」を説明する。What（何を）ではなく Why（なぜ）に焦点を当てる。箇条書き可。

フォーマット：
```
<prefix>: <件名>（#XX）

<なぜこの変更が必要か。背景・動機・解決する問題を書く>
```

関連 Issue が不明な場合はユーザーに確認してください。

## ステップ4：承認判断

**通常はユーザー確認なしで即コミットする。**

以下のいずれかに該当する場合のみ、コミット前にユーザーに確認を求めること：

- スコープ外の変更が含まれている疑いがある
- 破壊的変更（API の削除・リネーム、データ構造の変更など）を含む
- `.env`・認証情報・シークレットが誤って含まれている可能性がある
- 関連 Issue が特定できず、メッセージの方針が不明確

上記に該当しない場合はステップ5に直接進む。

## ステップ5：コミット実行

```bash
git commit -m "$(cat <<'EOF'
<prefix>: <件名>（#XX）

<本文>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

コミット後、`git log --oneline -3` で結果を確認してください。
