以下の手順で概念理解確認フェーズを実施してください。

## ステップ1：対象 diff の確認

引数でブランチが指定されている場合はそのブランチ、なければ `main..HEAD` を使用してください。

```bash
git diff main..HEAD  # ブランチ指定がある場合は main..<ブランチ名>
git diff main..HEAD --stat
```

diff の内容を把握し、以降のステップで質問を生成する素材にしてください。

## ステップ2：設計判断クイズ（2〜3問）

diff の中から以下の観点で**設計判断ポイント**を2〜3個抽出してください：

- なぜこのアプローチを選んだのか（別の実装方法との比較）
- この変更でトレードオフとなった要素
- 特定のパターン・命名・構造を選んだ理由

各質問を askuserinput で**1問ずつ**聞いてください。

**質問フォーマット**：

```
[設計判断クイズ 1/N]
ファイル: <file:line>

<コードの抜粋や変更内容を簡潔に提示>

質問: <「〇〇ではなく△△を選んだ理由は？」など具体的な選択肢を示す形で質問>

（スキップする場合は空白のまま送信してください）
```

## ステップ3：エッジケース質問（1〜2問）

変更されたロジックに対して、想定されるエッジケースを1〜2問 askuserinput で**1問ずつ**質問してください。

例：
- 「入力が空（null / undefined）の場合、この関数はどう動く？」
- 「同じ処理が並行して実行された場合の挙動は？」
- 「最大値・最小値の境界では何が起きる？」

**質問フォーマット**：

```
[エッジケース質問 1/N]
ファイル: <file:line>

質問: <エッジケースについての質問>

（スキップする場合は空白のまま送信してください）
```

## ステップ4：ソクラテス式フィードバック

各回答に対して以下の方針でフィードバックしてください：

| 回答の状態 | フィードバック方針 |
|-----------|-----------------|
| **正確な理解** | 「その通りです。」と認め、補足情報や関連する知識を提供する |
| **浅い・部分的な理解** | 直接答えを示さず、「〜という観点から考えるとどうでしょう？」とヒントで誘導する |
| **スキップ** | 正解と、考える際に参考になる観点を簡潔に説明する |

すべての質問への回答・フィードバックが終わったら、以下のフォーマットで**理解度総評**を出力してください：

```
## 理解度総評

**よく理解できていた点**:
- （具体的に）

**さらに深めると良い点**:
- （具体的に。責めるのではなく次のステップを示す）

**参考になるリソース（あれば）**:
- （ドキュメント・記事など）
```
