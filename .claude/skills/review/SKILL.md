以下の手順でセルフレビューを実施してください。

## ステップ1：変更内容の把握

対応する Issue 番号を確認してください（不明な場合はユーザーに確認）。

```bash
git log main..HEAD --oneline
git diff main..HEAD --stat
```

## ステップ2：スコープ外変更チェック

Issue のタイトル・やること と `git diff main..HEAD` を照合し、以下を確認してください：

- [ ] Issue のスコープ外のファイルが変更されていないか
- [ ] 意図しない変更（デバッグコード、無関係なリファクタ等）が混入していないか

スコープ外の変更が見つかった場合はユーザーに指摘し、別 Issue・別 PR に分けることを提案してください。

## ステップ3：影響範囲調査

差分レビューの**前に**、変更の影響範囲を調査してください。

### 3-1. 呼び出し元分析

変更された関数・export について、呼び出し元を検索してください：

```bash
# 変更されたファイルの export を特定し、プロジェクト内での参照箇所を検索
git diff main..HEAD --name-only | xargs grep -l 'export' 2>/dev/null
```

### 3-2. スキーマ/型変更検出

`src/domain/` 内の型・インターフェースが変更された場合、その型を参照しているファイルを列挙してください。

### 3-3. テスト影響評価

変更された `src/X.ts` に対応する `tests/X.test.ts` が更新されているか確認してください。未更新のテストファイルをフラグしてください。

### 出力フォーマット

```
| 変更ファイル | 呼び出し元 | テスト更新 | スキーマ変更 |
|-------------|-----------|-----------|-------------|
| src/...     | ...       | ✅ / ❌    | あり / なし  |
```

## ステップ4：プロジェクトルール読み込み

変更ファイルパスに基づき、`.claude/rules/` から該当するルールを読み込んでください：

| 条件 | 適用ルール |
|------|-----------|
| 常時 | `.claude/rules/tdd.md` |
| `src/domain/**` or `tests/domain/**` が変更 | `.claude/rules/domain-changes.md` |
| E2E 関連ファイルが変更 | `.claude/rules/e2e.md` |

各ルールファイルを読み込み、その基準をステップ5のレビュー観点に統合してください。

## ステップ5：2段階レビュー

### Stage 5-1：網羅的指摘

`git diff main..HEAD` の全変更を精査し、指摘を列挙してください。

**レビュー観点**：
- コード品質（バグ・セキュリティ・命名・可読性）
- ステップ4で読み込んだプロジェクトルールへの適合
- ステップ3の影響範囲（呼び出し元の更新漏れ、未テストのパス）
- 設計意図の整合（コードが Issue の目的に合っているか）

**出力フォーマット**（1指摘1行）：

```
- **[Critical/Warning/Info]** ファイル:行番号 — 指摘内容（根拠：ルール名 or ベストプラクティス）
```

### Stage 5-2：トリアージ

Stage 5-1 の指摘を以下の基準で分類し、対応方針を明示してください：

| レベル | 対応方針 |
|--------|---------|
| **Critical** | 必ず対応する（バグ・セキュリティ・ルール違反など） |
| **Warning** | できれば対応する（品質改善・潜在的問題など） |
| **Info** | YAGNI にならず、より良くなるなら対応する（nit・imo など） |

- **Critical が 0 件 → LGTM**：ステップ7に進む
- **Critical あり**：ステップ6へ

## ステップ6：修正ループ（振り子パターン検出付き）

Critical の指摘箇所を修正してください。

### 振り子パターン検出

修正をコミットする前に、以下を確認してください：

> 前回の修正イテレーションで変更した箇所を、今回の修正で元に戻していないか？
> ファイル X の行 Y を A→B に変更した後、B→A（または A と同等の状態）に戻そうとしている場合、それは振り子パターン（オシレーション）です。

**振り子パターンを検出した場合**：
- ループを即座に停止する
- 「振り子パターン検出：[ファイル:行] が A→B→A に変更されています。別のアプローチを検討してください。」とユーザーに報告する
- ユーザーの指示を仰ぐ

### 修正後の流れ

1. `/commit` の手順でコミットする（Warning/Info は判断して対応）
2. ステップ5に戻り再レビューする

> **ループ終了条件**：Stage 5-2 で Critical が 0 件になったとき。

## ステップ7：フォローアップチケット自動作成

Stage 5-2 の残存 Warning/Info の中から、以下の基準を**両方**満たすものを特定してください：

- 今の PR に含めると scope creep になる（現在の Issue スコープ外）
- 放置すると将来の技術的負債になる可能性がある

該当するものが**ある場合**、ユーザー入力なしに自律的に GitHub Issue として作成してください：

```bash
# 現在の Issue のラベルを確認する
gh issue view <Issue番号> --json labels --jq '.labels[].name'

# フォローアップ Issue を作成する（1件ずつ）
gh issue create \
  --title "フォローアップ: <指摘内容の要約>" \
  --label "<現在 Issue と同じラベル>" \
  --assignee "@me" \
  --body $'## 背景\n\n`/review` 実行時に指摘されたが、現在の PR スコープ外のため将来対応としたもの。\n\n**元の指摘**:\n- <指摘内容>（<ファイル:行番号>）\n\n## やること\n\n- [ ] <対応内容>\n\n## 完了条件（DoD）\n\n- [ ] <検証可能な完了基準>'

# Project に追加する
gh project item-add 6 --owner reiya0104 --url "<作成された Issue の URL>"
```

作成した Issue の URL を記録しておいてください（次のステップ8のサマリーに含めます）。

該当するものが**ない場合**は「フォローアップ Issue なし」としてスキップしてください。

## ステップ8：レビューサマリー出力

以下のフォーマットでサマリーを出力してください（`/create-pr` がこの出力を参照します）：

```
## レビューサマリー

**対応 Issue**: #XX
**影響範囲**:
- （ステップ3の結果を要約）

**適用ルール**: tdd.md, domain-changes.md, ...
**レビュー結果**: LGTM / N件の Warning 残存

**残存 Warning/Info**:
- （あれば列挙）

**フォローアップ Issue**:
- #YY タイトル（自動作成）
- （なければ「なし」）
```

### 継続的改善

レビュー中に、複数ファイルで同じパターンの指摘が繰り返された場合、`.claude/rules/` への新ルール追加を提案してください：

> 「繰り返し指摘パターン検出：[パターンの説明]。`.claude/rules/` に新ルールとして追加しますか？」

## ステップ9：次アクション選択

レビューサマリーを出力したら、askuserinput で次のアクションをユーザーに選択させてください：

```
レビューが完了しました。次のアクションを選択してください：

A) comprehension-check — diff の設計判断・エッジケースについて理解を深める質問フェーズに進む
B) create-pr — PR を作成する
```

- **A を選択した場合**：`/comprehension-check` スキルの手順を実行する
- **B を選択した場合**：`/create-pr` スキルの手順を実行する
